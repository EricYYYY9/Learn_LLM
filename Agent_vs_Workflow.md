## 一、一句话区别

> **Workflow 的"下一步做什么"是程序员写代码时定好的；Agent 的"下一步做什么"是 LLM 运行时决定的。**

把这句话刻在脑子里，下面所有内容都是它的展开。

| 维度 | Workflow | Agent |
|---|---|---|
| 谁决定流程？ | 程序员（编译期） | LLM（运行期） |
| 流程是什么？ | 代码里画好的 DAG / 状态机 | LLM 边做边推理出来的 |
| LLM 的角色 | 流程中**某些节点的处理器** | **整个流程的导演** |

## 二、一个生活类比：流水线工人 vs 解决问题的工程师

| 类比 | Workflow | Agent |
|---|---|---|
| 角色 | **流水线工人** | **解决问题的工程师** |
| 工作方式 | 老板（程序员）已经把动作写在 SOP 上：「拧螺丝 → 贴标签 → 装箱」，工人按顺序执行 | 老板只说「把这台坏机器修好」，工程师自己决定先拆哪里、用什么工具、要不要换零件 |
| 是否聪明 | 聪明也只在某一步用得上（比如"质检"那一步用 AI 视觉判断） | 聪明体现在**整个流程的决策**上 |
| 适合的活 | 大批量重复任务 | 没标准答案、需要随机应变的任务 |
| 失败时 | 卡在某一步、报错、人工介入 | Agent 自己换路径再试一次 |

> **理解关键**：Workflow 里 LLM 是"工人"，Agent 里 LLM 是"工程师"。

![Workflow vs Agent 控制流对比](assets/workflow_vs_agent_flow.png)

图中两点最关键：

1. **左边 Workflow**：流程线（紫色 → 蓝色 → 绿色菱形 → ...）是**预先画好的**，LLM 节点（蓝色框）只是其中一两个"智能岗位"，连"是 / 否"分支也是程序员写在代码里的 `if`。
2. **右边 Agent**：每一个绿色"工具调用"框前面都有一个黄色"LLM 思考"框——**每一步都先问 LLM"下一步做什么"**，路径不是预先画好的，而是当场推理出来的。

> 一句话：左图的箭头是开发者画的，右图的箭头是 LLM 画的。

## 四、用同一个"退款"任务对比

任务：用户提交了一笔退款工单，系统要处理它。

### Workflow 视角（程序员预先定好流程）

```
[用户提交工单]
       │
       ▼
[LLM 提取信息]  ← LLM 在这里只做"信息抽取"
       │
       ▼
[查订单数据库]
       │
       ▼
[LLM 判断是否符合政策]  ← LLM 在这里只输出"是 / 否"
       │
   ┌───┴───┐
   是      否              ← 分支由代码 if/else 决定
   │       │
   ▼       ▼
[执行退款] [生成拒绝邮件]
   │       │
   └───┬───┘
       ▼
   [发送通知]
```

特征：
- LLM 总共被调 **2 次**（抽取 + 判断），每次任务都一样多
- 每个箭头都是 `if / for / function call` 写死的
- 同一类工单永远走同一条路径

### Agent 视角（LLM 自己决定每一步）

```
[用户：处理这个退款]
       │
       ▼
[LLM 思考：我需要先了解申请内容]
       │
       ▼
[调用 read_ticket()]
       │
       ▼
[LLM 思考：高价值订单，需要查特殊政策]
       │
       ▼
[调用 search_policy("高价退款")]
       │
       ▼
[LLM 思考：政策允许，但需主管审批]
       │
       ▼
[调用 create_approval()]
       │
       ▼
[LLM 思考：审批通过，执行退款]
       │
       ▼
[调用 process_refund() + send_email()]
```

特征：
- LLM 被调 **N 次**（每次决策一次），N 取决于任务复杂度
- 没有任何一个箭头是程序员写死的，都是 LLM 推理出来的
- 同一个 Agent 面对**不同工单**会走**完全不同的路径**——比如普通退款可能直接调 `process_refund()`、跳过审批；可疑退款可能多调几次 `search_user_history()`

## 五、用代码对比（最直观）

### Workflow 写法

```python
def handle_refund(ticket):
    info = llm.extract(ticket)               # ① LLM 节点：抽取
    order = db.query(info.order_id)          # ② 普通代码

    if llm.judge(info, order, policy):       # ③ LLM 节点：判断
        refund(order)                        # ④ 代码 if 分支
    else:
        send_reject_email(info)              # ⑤ 代码 else 分支

    notify(info.user_id)                     # ⑥ 普通代码
```

**核心特征**：流程结构（`if/else`、调用顺序）是程序员写的；LLM 只在 `①③` 出现，是"被调用的工具"。

### Agent 写法

```python
def handle_refund(ticket):
    state = {"goal": f"处理工单 {ticket.id}", "history": []}

    while not state.get("done"):
        action = llm.decide_next_step(state)   # ★ LLM 决定下一步做什么
        if action.type == "FINISH":
            break
        result = tools.run(action.name, action.args)  # 执行 LLM 选的工具
        state["history"].append({"action": action, "result": result})
```

**核心特征**：流程结构是一个 `while + LLM.decide` 循环；**LLM 反过来调度代码**——`llm.decide_next_step` 返回什么，代码就调什么工具。

> **看这两段代码就能秒懂区别**：
> - Workflow：`llm.xxx()` 出现在固定位置，代码是主导。
> - Agent：`llm.decide_next_step()` 决定调用什么 `tool.xxx()`，LLM 是主导。

## 六、量化对比表

| 维度 | Workflow | Agent |
|---|---|---|
| **控制者** | 代码 / 开发者 | LLM |
| **LLM 调用次数** | 固定（每个流程 1~3 次） | 不固定，随任务复杂度 N 次（典型 3~10 次） |
| **Token 消耗** | 低（约 1×） | 高（约 3~10×，与步骤数成正比） |
| **可预测性** | 高（同输入 → 同路径） | 低（同输入可能走不同路径） |
| **灵活性** | 低（流程改动需改代码） | 高（改 prompt 或工具列表即可） |
| **延迟** | 低（步骤少且并行容易） | 高（多轮串行 LLM 调用） |
| **调试难度** | 容易（每步可单测） | 困难（路径不确定，需要 trace） |
| **适合任务** | 流程已知、规则清晰 | 流程未知、需要随机应变 |
| **典型场景** | 订单处理、报表生成、审批流 | 深度调研、Coding Agent、复杂客服工单分析 |

> **修正提示**：网上常见的"Agent token 4~8×"是个粗略数字。实际倍率 ≈ **(决策步骤数 + 1) × 平均 prompt 长度倍率**，复杂任务可能 20×+。面试时给"3~10×"+"与步骤数成正比"的解释比给具体数字更稳。

## 七、什么时候用哪个？决策树

```
         你的任务路径是不是已知 & 稳定？
                       │
        ┌──────────────┴──────────────┐
        是                              否
        │                              │
        ▼                              ▼
   用 Workflow            状态空间是不是开放（可能涉及未见过的子任务）？
   （把 LLM 当节点）                    │
                       ┌────────────────┴───────────────┐
                       否                                 是
                       │                                  │
                       ▼                                  ▼
              主流程 Workflow              用 Agent
              + 异常分支用 Agent           （LLM 主导规划）
              （混合架构）
```

经验法则：

- 能用 Workflow 解决就**别用 Agent**——便宜、可控、可测试。
- 只在以下情况上 Agent：
  1. 任务路径不可预测（深度研究、Coding 任务、复杂排障）
  2. 工具数量大且组合多（10+ 工具，写 if/else 不现实）
  3. 用户输入开放（"帮我分析这家公司"这种没有标准 SOP）

## 八、生产中最常见的姿势：混合架构

> 别把 Workflow 和 Agent 对立起来。**绝大多数生产系统是 Workflow + Agent 的混合**，这也是 Anthropic 在《Building effective agents》里的官方建议。

模式示意：

```
              ┌──────────── 入口路由 ────────────┐
              │  (Workflow + LLM 分类器)         │
              └──────┬───────────────┬───────────┘
                     │               │
            简单 / 高频              复杂 / 长尾
                     │               │
                     ▼               ▼
            ┌──────────────┐  ┌──────────────┐
            │  Workflow    │  │   Agent      │
            │  (固定路径)  │  │ (LLM 自主规划)│
            └──────┬───────┘  └──────┬───────┘
                   │                 │
                   └────────┬────────┘
                            ▼
                       人工兜底
                  (置信度低 / 失败时升级)
```

智能客服为例：

| 工单类型 | 用什么 | 为什么 |
|---|---|---|
| 「我想查我的物流」 | Workflow | 路径固定：查单号 → 查物流 API → 回复 |
| 「我对这件商品的尺寸有疑问」 | Workflow + RAG | 检索知识库 → LLM 总结回复 |
| 「我退货后又想换尺码，但已超期，订单还分了赠品」 | Agent | 涉及多策略组合，需要 LLM 自主拆解 |
| 任何用户明确表达"投诉" | 直接转人工 | 风险敏感，不让 AI 决策 |

**好处**：
- 90% 简单流量走 Workflow，**省 token、低延迟、可测试**
- 10% 长尾流量走 Agent，**保灵活性**
- 关键风险点直接转人工，**保安全**

## 九、原文逐句校正（含面试加分点）

> 你给的原文整体方向正确，下面是几处建议优化的地方：

| 原文表述 | 问题 | 校正建议 |
|---|---|---|
| "Workflow 就是把流程提前写死在代码里" | "写死"略贬义，容易让人误以为 Workflow 是不好的方案 | 改为"流程在编码期已经确定下来"，并补充"这正是 Workflow 的优势：可预测、可测试" |
| "LLM 只是其中某些节点的处理器" | 准确，但可加一句对比 | 补一句"它**不决定下一步走哪条边**" 让对比更鲜明 |
| "Agent 接收到目标，自主规划执行路径" | 准确 | 加一句"每一轮都要再调一次 LLM 决定下一步"——突出**多轮调用**这个 Agent 的本质代价 |
| "Token 消耗 高（约 4-8×）" | 数字过于具体，易被追问 | 改为"3~10×，且与决策步骤数成正比"；本质是 N+1 次 LLM 调用 |
| "典型场景：研究分析 / 客服" | "客服"过笼统——大部分客服是 Workflow | 改为"研究分析 / Coding Agent / 长尾复杂工单"；普通客服仍以 Workflow 为主 |
| "调试难度 容易 vs 困难" | 准确，但可补一句**为什么** | 补："Agent 路径不确定，需要 trace + replay 工具（如 LangSmith）才能调试" |
| "混合架构" 段落 | 准确，是面试加分点 | 强调一句"Anthropic 官方文档明确推荐：能用 Workflow 就别上 Agent" |

**面试场景下原文最容易被追问的两个点**：

1. **"为什么不全用 Agent？"** —— 答：成本（token × N 次）、延迟、不可预测、难调试、安全风险
2. **"什么时候必须用 Agent？"** —— 答：路径不可预测 + 工具数量大 + 输入开放 这三点同时成立

## 十、面试一句话答法

> "Workflow 和 Agent 的根本区别是**控制权归属**：Workflow 的控制流是程序员在编码期固化的代码 DAG，LLM 只是某些节点的处理器；Agent 的控制流是 LLM 在运行期推理出来的，每一步都重新调 LLM 决定下一步。所以 Workflow 可预测、便宜、易测试，但不能应对开放任务；Agent 灵活、能应对长尾，但 token 成本是 Workflow 的 N 倍且难调试。**生产中最佳实践不是二选一，而是 Workflow 提供主干 + Agent 处理长尾 + 关键节点人工兜底的混合架构**——这也是 Anthropic 在《Building effective agents》里的官方建议。"
